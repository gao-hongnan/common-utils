How WiqW^{q}_iWiq​ is implemented in practice?

The notation WiqW^{q}_iWiq​ is used in the paper to denote the weight matrix for the queries (Q) of the iii-th head. However, it's essential to understand how this is implemented in practice.

The entire process can be seen as a two-step operation:

  1. Apply Linear Transformations: You apply linear transformations to the whole embeddings to create larger matrices for Q, K, V. These matrices have dimensions that account for all heads. In practice, this can be implemented using a single linear layer, such as:

    Q=embeddings@Wq Q = \text{{embeddings}} @ \mathbf{W}^q Q=embeddings@Wq

    where Wq\mathbf{W}^qWq has dimensions D×(h⋅dq)D \times (h \cdot d_q)D×(h⋅dq​).

  2. Reshape and Split: After applying the linear transformations, you reshape and split the result into individual heads. The reshaping ensures that the final dimensions are [N,H,S,dq][N, H, S, d_q][N,H,S,dq​], where NNN is the batch size, HHH is the number of heads, SSS is the sequence length, and dqd_qdq​ is the dimension of queries per head.

So, while the paper uses notation like WiqW^{q}_iWiq​, this doesn't mean that you directly apply a different linear transformation to different parts of the embeddings. Instead, you apply a single large linear transformation to the whole embeddings and then reshape the result to obtain the individual heads.

In mathematical terms, the overall operation can be seen as:

Qall heads=embeddings@WqQhead i=Qall heads[:,i⋅dq:(i+1)⋅dq] \begin{align*} Q_{\text{{all heads}}} & = \text{{embeddings}} @ \mathbf{W}^q \\ Q_{\text{{head i}}} & = Q_{\text{{all heads}}}[:, i \cdot d_q : (i + 1) \cdot d_q] \end{align*} Qall heads​Qhead i​​=embeddings@Wq=Qall heads​[:,i⋅dq​:(i+1)⋅dq​]​

Here, Qall headsQ_{\text{{all heads}}}Qall heads​ is the result of applying the linear transformation, and Qhead iQ_{\text{{head i}}}Qhead i​ is the portion corresponding to the iii-th head, obtained by slicing along the last dimension.

HEad is similar to kernels in CNN

The multi-head attention mechanism is similar to the convolutional layer in convolutional neural networks. In a convolutional layer, you apply multiple kernels to the input to obtain multiple feature maps. Similarly, in the multi-head attention mechanism, you apply multiple attention heads to the input to obtain multiple output vectors.

so the catch is you do not split the embeddings in H heads, instead you split the linear transformed embeddings?

# Apply linear transformations to compute Q, K, V
# NOTE: here is an important misconception that if you have
# 8 heads, then you SPLIT the embeddings into 8 parts and
# then apply linear transformations to each part. This is
# WRONG. You apply linear transformations to the whole
# embeddings and then split the result into 8 parts.

You don't split the original embeddings into HHH heads; instead, you apply linear transformations to the original embeddings and then split the transformed embeddings into HHH heads.

Here's the step-by-step process again, highlighting this specific aspect:

  1. Apply linear transformations for queries, keys, and values to the entire embeddings, creating matrices Q,K,Q, K,Q,K, and VVV.
  2. Split these transformed matrices into HHH different heads, each having lower dimensions (e.g., if the original dimension is 512 and there are 8 heads, each head will have a dimension of 64).
  3. Process each head through the Scaled Dot-Product Attention mechanism.
  4. Concatenate the outputs from all the heads and pass through a final linear layer.

The split after the linear transformations allows the model to create multiple different projections of the input and process them independently. This enables the model to focus on different aspects of the input across the different heads, enhancing its ability to model complex relationships.

Confusion on Weight matrix per head

The notation and explanation in the original papers and many articles do indeed mention separate weight matrices for each head, such as WiQW^{Q}_iWiQ​, but in implementation, it's common to represent these separate weights within a single large weight matrix. The notation might be different, but the mathematical operation is equivalent.

Here's how the two approaches relate:

  1. Separate Weight Matrices Notation (Paper Notation): In the theoretical description, you can imagine having separate weight matrices WiQW^{Q}_iWiQ​ for each head, and then you multiply the input embeddings by each weight matrix, applying the transformation separately for each head.

  2. Single Large Weight Matrix Implementation (Your Code): In your implementation, you create one large weight matrix, WqW_qWq​, that combines all the individual weight matrices for each head. When you multiply the input embeddings by WqW_qWq​, you create a large transformed matrix. Then, by slicing this large matrix, you separate it into HHH different heads, effectively applying the individual weight matrices WiQW^{Q}_iWiQ​ for each head.

The two approaches are mathematically equivalent. In the second approach, the separate weight matrices for each head are not explicitly defined as learnable parameters in the code. Instead, they are implicitly represented within the single large weight matrix WqW_qWq​ and separated by slicing after the linear transformation.

This approach can be more efficient computationally and often aligns better with hardware and library optimizations, but it may create confusion when comparing to the paper's notation. The key is to understand that the mathematical relationships and learning dynamics are the same, even though the notation and coding structure might differ.

Approach 1: Single Large Weight Matrix Implementation (Paper's Code)

In this approach, we concatenate all the individual weight matrices WhqW^{q}_{h} Whq​ into one large weight matrix WqW_qWq​:

Wq=[w1,1qw1,2q…w1,Dqw2,1qw2,2q…w2,Dq⋮⋮⋱⋮wD,1qwD,2q…wD,Dq]D×D=[W1qW2q…WHq]∈RD×D \begin{aligned} W_q = \begin{bmatrix} w^{q}_{1,1} & w^{q}_{1,2} & \ldots & w^{q}_{1,D} \\ w^{q}_{2,1} & w^{q}_{2,2} & \ldots & w^{q}_{2,D} \\ \vdots & \vdots & \ddots & \vdots \\ w^{q}_{D,1} & w^{q}_{D,2} & \ldots & w^{q}_{D,D} \end{bmatrix}_{D \times D} &= \begin{bmatrix} W^{q}_1 & W^{q}_2 & \ldots & W^{q}_H \end{bmatrix} \in \mathbb{R}^{D \times D} \end{aligned} Wq​=⎣⎡​w1,1q​w2,1q​⋮wD,1q​​w1,2q​w2,2q​⋮wD,2q​​……⋱…​w1,Dq​w2,Dq​⋮wD,Dq​​⎦⎤​D×D​​=[W1q​​W2q​​…​WHq​​]∈RD×D​

where each WhqW^{q}_hWhq​ is a matrix with dimensions D×DH=D×dqD \times \frac{D}{H} = D \times d_qD×HD​=D×dq​.

In other words, if the embedding dimension is 512 and there are 8 heads, the original WqW^qWq matrix is of size 512×512512 \times 512512×512, and we can decompose it into 8 matrices of size 512×64512 \times 64512×64, each forming a column of the original matrix.

We then multiply the embeddings by this large weight matrix:

Q=Z⋅Wq Q = \mathbf{Z} \cdot W^{q} Q=Z⋅Wq

Then, we slice the result into HHH parts:

Qh∈RB×L×dq=Q[:,:,h⋅DH:(h+1)⋅DH]=Q[:,:,h⋅dq:(h+1)⋅dq]=Z⋅Whq \begin{aligned} Q_{h} \in \mathbb{R}^{B \times L \times d_q} &= Q\left[:, :, h \cdot \frac{D}{H} : (h+1) \cdot \frac{D}{H}\right] \\ &= Q\left[:, :, h \cdot d_q : (h+1) \cdot d_q\right] \\ &= \mathbf{Z} \cdot W^{q}_{h} \end{aligned} Qh​∈RB×L×dq​​=Q[:,:,h⋅HD​:(h+1)⋅HD​]=Q[:,:,h⋅dq​:(h+1)⋅dq​]=Z⋅Whq​​

where WhqW^{q}_{h}Whq​ is the submatrix of WqW^{q}Wq that corresponds to the hhh-th head, or in other words, let's say W1qW^{q}_1W1q​, the first head, it means subsetting the WqW^qWq with rows dimension unchanged (i.e. 512), and taking the first 64 columns, resulting in a matrix of size 512×64512 \times 64512×64.

Approach 2: Separate Weight Matrices Notation (Paper Notation)

Suppose we have HHH heads and our embedding matrix Z\mathbf{Z}Z has dimensions B×L×DB \times L \times DB×L×D, where BBB is the batch size, LLL is the sequence length, and DDD is the embedding dimension.

For each head hhh, we have a weight matrix WhqW^{q}_{h}Whq​ with dimensions D×DH=D×dqD \times \frac{D}{H} = D \times d_qD×HD​=D×dq​, and we apply this transformation to the embeddings:

Qh∈RB×L×dq=Z⋅Whq Q_{h} \in \mathbb{R}^{B \times L \times d_q} = \mathbf{Z} \cdot W^{q}_{h} Qh​∈RB×L×dq​=Z⋅Whq​

So the confusion arises because in the code implementation we do not see an explicit definition of the separate weight matrices WhqW^{q}_{h}Whq​, but they are implicitly represented within the single large weight matrix WqW^{q}Wq. But actually you can see from approach 1, the WqW^qWq is just a concatenation of all the WhqW^{q}_{h}Whq​, so it's just a different way of representing the same thing.

is the FFN in encoder just a MLP layer

Yes, the Feed-Forward Network (FFN) in the Transformer's encoder is essentially a Multi-Layer Perceptron (MLP) layer. It typically consists of two fully connected layers, with a non-linear activation function (usually ReLU) applied after the first layer.

Here's the general structure of the FFN in the Transformer's encoder:

  1. First Linear Layer: The input is passed through a fully connected linear layer with weight matrix W1W_1W1​ and bias b1b_1b1​.
  2. Activation Function: A non-linear activation function (such as ReLU) is applied to the result of the first linear layer.
  3. Second Linear Layer: The activated output is then passed through another fully connected linear layer with weight matrix W2W_2W2​ and bias b2b_2b2​.
  4. Optional Dropout: Some implementations might include dropout for regularization after one or both of the linear layers.

The mathematical expression for this process would look something like:

FFN(x)=W2⋅ReLU(W1⋅x+b1)+b2 \text{FFN}(x) = W_2 \cdot \text{ReLU}(W_1 \cdot x + b_1) + b_2 FFN(x)=W2​⋅ReLU(W1​⋅x+b1​)+b2​

Where xxx is the input to the FFN, and W1W_1W1​, W2W_2W2​, b1b_1b1​, and b2b_2b2​ are learnable parameters.

So, the FFN in the Transformer's encoder is effectively a specific form of a Multi-Layer Perceptron with two layers, with the goal of learning position-wise transformations of the input.
